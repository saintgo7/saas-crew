# WKU 소프트웨어 Crew Project - 리스크 관리

## 1. 리스크 관리 프레임워크

### 1.1 리스크 관리 프로세스

```
리스크 식별 → 리스크 분석 → 리스크 평가 → 리스크 대응 → 모니터링
```

### 1.2 리스크 매트릭스

| 영향도 \ 확률 | 낮음 (1) | 중간 (2) | 높음 (3) |
|---------------|----------|----------|----------|
| **높음 (3)**  | 중간 (3) | 높음 (6) | 매우 높음 (9) |
| **중간 (2)**  | 낮음 (2) | 중간 (4) | 높음 (6) |
| **낮음 (1)**  | 매우 낮음 (1) | 낮음 (2) | 중간 (3) |

**우선순위**:
- 9: 즉시 대응 (Critical)
- 6-8: 조기 대응 (High)
- 3-5: 모니터링 (Medium)
- 1-2: 수용 (Low)

## 2. 기술적 리스크

### 리스크 T-1: 클라우드 IDE 성능 이슈

**설명**: 동시 사용자 증가 시 IDE 반응 속도 저하

**확률**: 중간 (2)
**영향도**: 높음 (3)
**리스크 점수**: 6 (High)

**원인**:
- 서버 리소스 부족
- 네트워크 레이턴시
- Docker 컨테이너 오버헤드

**영향**:
- 사용자 경험 저하
- 이탈률 증가
- 부정적 리뷰

**대응 전략**:
1. **예방 (Mitigation)**
   - 조기 부하 테스트 (100+ 동시 사용자)
   - 컨테이너 최적화
   - CDN 활용
   - 지역별 서버 배치

2. **대안 (Contingency)**
   - 경량 에디터 제공 (CodeMirror)
   - 로컬 개발 환경 가이드
   - 사용 시간대 분산 유도

3. **전환 (Fallback)**
   - 외부 IDE 서비스 연동 (GitPod, CodeSandbox)

**모니터링 지표**:
- 평균 응답 시간 (< 200ms 목표)
- 에러율 (< 1%)
- 사용자 만족도 (> 4.0/5.0)

**담당자**: CTO
**재검토 주기**: 주간

## 3. 비즈니스 리스크

### 리스크 B-1: 사용자 획득 부진

**설명**: 목표 사용자 수 미달 (1년차 500명 목표)

**확률**: 중간 (2)
**영향도**: 높음 (3)
**리스크 점수**: 6 (High)

**원인**:
- 마케팅 예산 부족
- 제품 매력도 부족
- 경쟁 심화
- 학생 관심도 낮음

**영향**:
- 수익 목표 미달
- 투자 유치 어려움
- 팀 사기 저하

**대응 전략**:
1. **예방**
   - PMF (Product-Market Fit) 검증
   - 베타 테스트 피드백 반영
   - 바이럴 마케팅 강화
   - 학과 공식 채택

2. **조기 경보**
   - 주간 신규 가입자 추적
   - 목표 대비 진행률 (< 80% 시 알림)

3. **대응**
   - 마케팅 예산 증액
   - 무료 프로모션 확대
   - 인플루언서 협업

**담당자**: CMO
**재검토 주기**: 주간

### 리스크 B-2: 높은 이탈률 (Churn)

**설명**: 사용자 가입 후 1개월 내 이탈 > 50%

**확률**: 높음 (3)
**영향도**: 중간 (2)
**리스크 점수**: 6 (High)

**원인**:
- 온보딩 복잡
- 콘텐츠 부족
- 학습 동기 부여 실패
- 버그 및 성능 이슈

**대응 전략**:
1. **예방**
   - 온보딩 최적화 (5분 이내 첫 성공 경험)
   - 게이미피케이션 (배지, 스트릭)
   - 이메일/푸시 알림
   - 스터디 그룹 매칭

2. **탐지**
   - 코호트 분석
   - 이탈 예측 모델
   - 사용자 행동 추적

3. **복구**
   - 윈백 캠페인 (Win-back)
   - 설문조사 (이탈 이유)
   - 인센티브 (복귀 시 혜택)

**담당자**: CPO + CMO
**재검토 주기**: 주간

### 리스크 F-1: 자금 고갈 (Runway 부족)

**설명**: 예상보다 빠른 현금 소진

**확률**: 중간 (2)
**영향도**: 매우 높음 (3)

- 수익 부진
- 비용 초과 (특히 인프라)

**영향**:

- 직원 급여 지급 불가

**대응 전략**:

   - 다양한 수익원 (B2B, 정부 지원)

2. **조기 경보**
   - 월간 현금 흐름 추적
   - Runway < 12개월 시 알림

3. **비상 대응**
   - 비용 긴축 (비필수 지출 중단)
   - 추가 투자 유치
   - 브릿지 론 (교량 대출)

## 11. 위기 관리 계획 (Crisis Management)

### 11.1 위기 레벨 정의

**레벨 1 (Minor)**: 일부 기능 장애, 일부 사용자 영향
**레벨 2 (Moderate)**: 주요 기능 장애, 다수 사용자 영향
**레벨 3 (Major)**: 서비스 전체 중단
**레벨 4 (Critical)**: 데이터 유실, 보안 사고

### 11.2 위기 대응 팀

**위기 관리자**: CEO
**기술 리드**: CTO
**커뮤니케이션 리드**: CMO
**고객 지원 리드**: COO

### 11.3 위기 대응 프로토콜

1. **탐지 및 알림** (5분 내)
   - 자동 모니터링 알림
   - 사용자 신고

2. **위기 팀 소집** (15분 내)
   - Slack #crisis 채널
   - 긴급 회의

3. **영향 평가** (30분 내)
   - 영향 범위 파악
   - 레벨 결정

4. **대응 실행** (즉시)
   - 기술적 대응
   - 사용자 커뮤니케이션
   - 언론 대응 (필요시)

5. **복구 및 사후 조치** (24-48시간)
   - 서비스 복구
   - 사후 분석 (Post-Mortem)
   - 재발 방지 대책
